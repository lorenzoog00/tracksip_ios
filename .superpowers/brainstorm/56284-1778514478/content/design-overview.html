Design: Food-Aware BAC — Hybrid Approach

3 components working together. Does this architecture look right?

1 — Pre-Event Notification

A push notification fires ~45 min before the user's historically typical drinking start time (derived from past event data). No action required — it's a nudge, not a prompt. If the user has no history yet, it fires at a configurable time (default 8:00 PM on Friday/Saturday).

2 — Event Start Prompt

When the user starts a new event, a single extra step asks "How full is your stomach right now?" — three options, one tap. This sets the initial absorption baseline for the entire BAC model.

3 — In-Event Food Logging

During the night, users can log food the same way they log water — one tap in the active event screen. Each log timestamps the entry and temporarily boosts the stomach fullness level, which then decays over 2–3 hours (gastric emptying model).

BAC Model Impact

  • Empty stomach: baseline (current behavior) — fastest absorption
  • Snack: ~15% peak BAC reduction, 15 min absorption delay
  • Full Meal: ~30% peak BAC reduction, 30–45 min absorption delay
  • Decay: fullness effect reduces linearly over 2–3 hrs back to Empty
  • In-event food log: resets the decay clock from that moment forward
