3 approaches to food-aware BAC

A push notification reminds you to eat before going out. Here's how to model food's effect on BAC calculations.

A

Stomach State Modifier

Set a stomach state at event start (Empty / Snack / Full Meal) and update it if you eat during the night. Each state applies a fixed absorption multiplier — Full Meal reduces peak BAC ~30%, Snack ~15%.

Pros

  • Simple to understand
  • Fast to log
  • Easy to implement

Cons

  • Ignores timing — eating at 8pm then drinking at 11pm treated the same as eating at 10pm
B

Time-Decaying Food Effect ★ Recommended

Each food log is timestamped. Food's protective effect decays over ~2–3 hours (modelling gastric emptying). BAC at any moment accounts for all food entries and how long ago they were eaten.

Pros

  • Most accurate
  • Timing-aware — old food counts less
  • Same quick-log UX as water entries

Cons

  • More complex model to tune
C

Hybrid — Baseline + Live Adjust

Notification primes the user. Event start asks stomach state (sets baseline). During the night, food logs boost fullness back up. State decays over time automatically. Middle ground between A and B.

Pros

  • Good accuracy
  • Baseline covers pre-event eating
  • Digestible UX

Cons

  • Two-step (notification + event start prompt) adds a little friction
